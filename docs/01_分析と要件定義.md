# グルメキャリー版 スカウトメール自動生成拡張機能 ― 分析 & 要件定義

対象サイト: `https://kanri.gourmetcaree.jp/shop-pc/member/detail/index/{member_id}`
ベースとなる既存資産: リクナビHRTech版 Chrome拡張 (`scout-rikunabi-main`)
使用プロンプト: `revised_prompt_v9.0.yaml`（資さんうどん・古川版）

---

## 1. 既存の仕組みの分析（リクナビHRTech版）

### 1.1 全体アーキテクチャ

Chrome拡張（Manifest V3）で、**採用管理サイト** と **Gemini** の2つのタブを
バックグラウンドのService Workerが仲介して連携させる構成。

```
┌────────────────────────┐      openGemini      ┌──────────────────┐
│ リクナビHRTech タブ    │  ─────────────────▶ │  background.js    │
│ content_rikunabi.js    │                      │ (Service Worker) │
│  ・候補者情報を抽出    │  ◀───────────────── │  ・タブ間の仲介   │
│  ・フォームへ自動入力  │      showResult      │  ・状態を保持     │
└────────────────────────┘                      └──────────────────┘
        ▲  MAIN world                                    │ fillPrompt
        │  (React setter)                                ▼
 content_rikunabi_main.js                        ┌──────────────────┐
                                                 │  Gemini タブ      │
                                                 │ content_gemini.js │
                                                 │  ・プロンプト投入 │
                                                 │  ・回答を待機取得 │
                                                 └──────────────────┘
                                                        ▲ MAIN world
                                                        │ (Quill操作)
                                                 content_gemini_main.js
```

### 1.2 ファイル構成と役割

| ファイル | World | 役割 |
|---|---|---|
| `manifest.json` | ― | 権限・content_scripts・対象URLの定義 |
| `background.js` | SW | 2タブ間のメッセージ中継、`pendingRequest` の状態管理、Geminiタブの生成/再利用 |
| `content_rikunabi.js` | isolated | **採用サイト側の中核**。候補者情報抽出／フローティングボタンUI／生成結果のクリーニング／フォーム自動入力／「確認」ボタン押下／送信カウンタ。**システムプロンプトを内蔵** |
| `content_rikunabi_main.js` | MAIN | Reactが管理する input/textarea に、ネイティブsetter経由で値を注入（React内部stateを正しく更新するため） |
| `content_gemini.js` | isolated | **Gemini側の中核**。入力エリア探索／送信ボタン押下／回答の安定待機（0.4秒×5回同一で完了判定、120秒タイムアウト）／件名・本文パース |
| `content_gemini_main.js` | MAIN | GeminiのQuillエディタ実体(`__quill`)に全文を投入し、Angularの変更検知を発火 |
| `popup.html` | ― | 使い方の説明のみ（機能ロジックなし） |
| `styles.css` | ― | フローティングボタン・結果ボックス・バッジの見た目 |

### 1.3 処理フロー（1通の生成）

1. 採用サイトでスカウトフォームを開くと、DOM監視(`MutationObserver`)が
   フォーム存在を検知し、右下にフローティングボタンを表示。
2. ボタン押下 → `extractCandidateInfo()` がパネル/カード/ページから候補者情報を抽出。
3. 内蔵システムプロンプト + 候補者情報を結合し、`openGemini` を background へ送信。
4. background がGeminiタブを開く/再利用し、ロード完了後 `fillPrompt` を送信。
5. Gemini側がプロンプトを投入・送信し、回答を安定待機して取得・パース。
6. `geminiResponse` で件名・本文を採用サイトへ返送。
7. 採用サイト側が結果をクリーニング（ラベル除去・Gemini余計コメント除去・文字数チェック）し、
   フォームへ自動入力 → 「確認」ボタンを自動クリック → 送信カウンタ +1。

### 1.4 技術的なキモ（移植で必ず再現が必要な点）

- **React/Angular対応の値注入**: 単に `el.value = x` では内部stateが更新されないため、
  `HTMLInputElement/HTMLTextAreaElement.prototype` の native setter を呼び、
  `input`/`change`/`blur` を dispatch する（`content_rikunabi_main.js`）。
- **isolated と MAIN の二層構成**: `chrome.runtime` は isolated、フレームワーク内部への
  アクセス（Quill/React setter）は MAIN。両者は `window.postMessage` で通信。
- **回答完了の判定**: Geminiの生成が止まったかを、テキストが5回連続同一かで判定。
- **出力のクリーニング**: セクションラベル・「いかがでしょうか」等のAIコメントを正規表現で除去。
- **セレクタの緩さ（サイト依存部分）**: 候補者パネル・件名/本文フィールド・「確認」ボタンは
  クラス名の部分一致やラベル近傍探索といったヒューリスティックで特定している。
  → **ここがサイトごとに作り直しが必要な箇所。**

### 1.5 プロンプトについて【確定】

- リポジトリ内の `content_rikunabi.js` には v9.0 から更に発展した文面
  （「内定まで最短14日」「2030年400店舗」「入社後の実績」など）が**ハードコード**されている。
- **依頼者確認済み: この `content_rikunabi.js` 内蔵版が最新プロンプトであり、これを正とする。**
  （添付 `revised_prompt_v9.0.yaml` は旧版として参照のみ）

## 2. 対象サイト（グルメキャリー）の分析

### 2.1 現時点で判明していること

- URL: `https://kanri.gourmetcaree.jp/shop-pc/member/detail/index/{member_id}`
  → `member/detail` は**候補者（会員）詳細ページ**。末尾はmember_id。
- `kanri.` サブドメイン = 店舗（求人企業）向けの**管理コンソール**。
- 未ログインでアクセスすると **HTTP 403**（＝認証必須。外部から中身を取得不可）。

**依頼者提供の実画面テキスト（2候補者分）から判明した画面構造:**

- 候補者詳細ページに**スカウトフォームが同居**しており、1画面で以下が並ぶ:
  1. 候補者ヘッダ: 会員ID（長いランダム文字列）／職種／現職企業／年齢／
     現年収レンジ／経験社数／居住都道府県／出身校／レジュメ更新日
  2. アピールポイント（存在する場合）
  3. 職務要約（長文テキスト）
  4. スキル・保有資格／語学力
  5. 職歴（企業名・職種・期間・業態・顧客層・業務タグ・マネジメント規模）
  6. 学歴
  7. 希望条件（転職意欲・希望雇用形態・希望職種・希望年収・希望勤務地）
  8. メモ・備考／候補者評価／スカウト履歴
  9. **スカウトフォーム**:
     - **求人票**（必須・テンプレート選択。「※テンプレートの求人票・件名・本文が反映されます」
       → テンプレ選択で件名・本文が自動反映される仕様）
     - **件名**（必須・**70文字**上限・文字数カウンタ表示）
     - **本文**（必須・**4000文字**上限・文字数カウンタ表示）
- 件名70 / 本文4000 の上限は**リクナビ版の文字数チェックと同値** → 既存チェックをそのまま流用可。
- 候補者情報の情報量はリクナビ版より豊富（職務要約・業務タグが詳細）。

### 2.2 残る不明点（実装時に確定が必要）

| # | 不明点 | なぜ重要か | 状態 |
|---|---|---|---|
| A | フォームのDOM構造（セレクタ・name属性） | 件名/本文への自動入力に必須 | **未確定**（HTML提供待ち。テキスト構造は判明済み） |
| B | フレームワーク（React / 素のHTML / jQuery 等） | 値注入方式（native setter要否）が変わる | 未確定（両対応の実装で吸収可能） |
| C | 候補者情報の掲載項目 | プロンプトへ渡す抽出ロジック | **ほぼ確定**（§2.1の通り） |

## 3. 要件定義

### 3.1 目的

グルメキャリー管理コンソールの候補者詳細/スカウト画面上で、ボタン一つで
候補者情報を読み取り、Geminiに古川版プロンプトでスカウトメールを生成させ、
フォームへ自動入力するChrome拡張を、リクナビ版と同等の使用感で提供する。

### 3.2 機能要件

| ID | 要件 | リクナビ版からの流用可否 |
|---|---|---|
| F-1 | スカウト画面上にフローティング生成ボタンを表示 | ◎ ほぼ流用（表示トリガ条件のみ調整） |
| F-2 | 候補者情報の自動抽出 | △ **サイト専用に作り直し**（項目A・C） |
| F-3 | 古川版プロンプト（v9.0）＋候補者情報を結合しGeminiへ | ◎ 流用（プロンプト差し替え） |
| F-4 | Geminiタブ生成・プロンプト投入・回答取得 | ◎ そのまま流用 |
| F-5 | 回答のクリーニング（ラベル/コメント除去・文字数チェック） | ◎ 流用（禁止ワード等は微調整） |
| F-6 | 件名・本文をフォームへ自動入力 | △ **セレクタをサイト専用に**（項目A・B） |
| F-7 | 「確認」→「送信」導線の自動化 | △ **サイトの導線に合わせ調整**（項目D） |
| F-8 | 当日送信件数カウンタ（24時リセット）＋バッジ | ◎ そのまま流用 |
| F-9 | popupの使い方説明 | ◎ 文言のみ差し替え |
| F-10 | **個人情報の除外**: Geminiへ送信する候補者情報から、氏名・会員ID等の個人識別情報を除去（マスキング）してから送る | ★ 新規（リクナビ版に無い要件） |

> F-10 補足: 会員IDはラベル「会員ID：」に続く英数字列として除去可能。氏名等は
> 画面上の掲載位置が確定し次第、除去対象パターンを追加する。年齢・職歴などの
> 属性情報はメール生成に必要なため送信対象に残す。

### 3.3 非機能要件

- Manifest V3 / Chrome拡張。外部サーバ不要（Geminiタブ経由でLLM利用）。
- `host_permissions` を `https://kanri.gourmetcaree.jp/*` と `https://gemini.google.com/*` に限定。
- 候補者情報・生成文はローカル完結（送信カウンタのみ `chrome.storage.local`）。
- サイトのHTML変更に強いよう、セレクタは複数フォールバックを持たせる。

### 3.4 サイト移植で「変更が必要な箇所」一覧

1. `manifest.json` の `matches` / `host_permissions` → `kanri.gourmetcaree.jp`。
2. `content_rikunabi.js` 相当（→ `content_gourmet.js`）:
   - `isScoutFormPresent()` … グルメキャリーのフォーム判定条件。
   - `extractCandidateInfo()` … 候補者項目の抽出セレクタ。
   - `fillForm()` / セレクタ群 … 件名・本文フィールド特定。
   - `clickConfirmButton()` … 確認/送信ボタンの文言・導線。
   - システムプロンプト … `content_rikunabi.js` 内蔵の最新版をそのまま採用（確定済み）。
   - 候補者情報の個人情報マスキング処理を追加（F-10）。
3. `content_rikunabi_main.js` 相当 … フレームワークが素のHTMLなら不要になる可能性（項目B次第）。
4. Gemini側（`content_gemini*.js`）・`background.js` … **原則そのまま流用可能**。
5. `popup.html` / `styles.css` … 文言・タイトル差し替えのみ。

### 3.5 想定成果物

- `manifest.json`
- `background.js`（流用）
- `content_gourmet.js` / `content_gourmet_main.js`（サイト専用に新規）
- `content_gemini.js` / `content_gemini_main.js`（流用）
- `popup.html` / `styles.css`（文言調整）
- README（インストール手順・使い方）

## 4. 確認事項のステータス

- **Q1: 対象画面のHTML提供** … **一部解決**。実画面の全文テキスト（2候補者分）の提供により、
  画面の項目構成・フォーム構成（求人票／件名70／本文4000）は確定。
  セレクタ確定のための **outerHTML は引き続き提供が望ましい**（無ければ汎用実装＋実機調整）。
- **Q2: LLMバックエンド** … **Gemini（gemini.google.com）連携を継続**（デフォルト採用）。
- **Q3: プロンプトの版** … **確定**。`content_rikunabi.js` 内蔵の発展版
  （「内定まで最短14日」「2030年400店舗」「入社後実績」入り）が最新であることを依頼者が確認。
  これをグルメキャリー版にもそのまま搭載する。
- **Q4: 送信の自動化レベル** … リクナビ版と同等（入力→確認まで自動、最終送信は人間）をデフォルト採用。
- **追加要件（依頼者指示）**: Geminiへ送る候補者情報から**個人情報を除外**すること（F-10）。
